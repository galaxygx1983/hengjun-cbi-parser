# 协议机制

## 核心约束

⚠️ **ACK超时500ms是协议固定值，不可修改**

网络问题必须通过物理层解决，不能调整超时参数。

## 超时参数

| 参数 | 值 | 说明 |
|------|------|------|
| ACK超时 | 500ms | 固定 |
| 最大重试 | 3次 | 原始+2次重传 |
| 中断判定 | 1500ms | 3×500ms |
| 硬件故障重试 | 6秒 | DC2间隔 |
| CTC切换 | 3秒 | 主机-主机故障 |
| 联锁切换 | 5秒 | 主机-主机故障 |

## 序号机制

### 序号变量

| 变量 | 说明 | 初始值 |
|------|------|--------|
| 发送序号 | 数据帧编号 | DC3后=1 |
| 确认序号 | 确认已收帧 | DC3后=0 |

### 序号控制规则

1. **发送帧时**: 发送序号=发送序号变量，确认序号=确认序号变量
2. **收到ACK**: 确认序号=发送序号 → 发送序号变量+1
3. **未确认**: 不发送新帧，超时重传
4. **收到重复帧**: 确认序号=发送序号，不重复处理
5. **序号差≥2**: 判定通信中断

## 主备通信

### 通信路径

| 路径 | 功能 |
|------|------|
| CTC主机 ↔ 联锁主机 | 完整通信 |
| CTC主机 ↔ 联锁备机 | 仅监控 |
| CTC备机 ↔ 联锁主机 | 查询，禁止控制 |
| 备机 ↔ 备机 | 最小通信 |

### 帧类型限制

- **BCC/TSD/ACA**: 仅CTC主机→联锁主机
- **SDI/SDCI**: 仅联锁主机→CTC主机
- **FIR/ACQ**: 仅联锁主机发送
- **TSQ**: 仅联锁主机发送

## 故障切换

### 触发条件
- **仅主机-主机故障时切换**
- CTC: 中断3秒后切换
- 联锁: 中断5秒后切换

### 切换流程
```
CTC主机故障 → 3秒超时 → 检查备机能否通信 → 切换
联锁主机故障 → 5秒超时 → 检查备机能否通信 → 切换
```

## 故障恢复

| 阶段 | 处理 |
|------|------|
| 检测中断 | 连续3次无ACK(1500ms) |
| CTC响应 | 发送DC2最多3次 |
| 判定故障 | 3次DC2无响应 |
| 恢复模式 | 每6秒发送DC2尝试恢复 |

## 序号异常处理

### 发送方校验

| 对比结果 | 含义 | 处理 |
|----------|------|------|
| 确认序号=发送序号 | 已确认 | 发送序号变量+1，可发下一帧 |
| 确认序号≠发送序号 | 未确认 | 丢弃该ACK，可能需重传 |

### 接收方校验

| 差值 | 含义 | 处理 |
|------|------|------|
| =1 | 预期帧 | 确认序号+1，正常处理 |
| =0 | 重复帧 | 发送ACK，**不重复处理** |
| ≥2 | 帧丢失 | 判定通信中断 |

### 序号交互异常场景

**CTC视角**：发送BCC(序号=1C)，收到SDCI确认序号=1B → BCC未确认，需重发

**CBI视角**：发送SDCI(序号=F3)，先收到BCC确认序号=F2(丢弃)，后收到ACK确认序号=F3 → 序号对齐

### 工程实践规则

1. 收到帧但确认序号≠自身发送序号 → 直接丢弃，不解析不反馈
2. 收到帧且确认序号=自身发送序号 → 判定正确应答，更新序号，终止超时
3. 500ms内无正确应答 → 重发原帧（仅更新确认序号+CRC），重发2次仍失败→通信中断

## 4路通信矩阵

| 通信路径 | 级别 | BCC | SDI/SDCI | FIR | RSR |
|----------|------|-----|-----------|-----|-----|
| 主机↔主机 | 完整 | ✅ | ✅ | ✅ | ✅ |
| 主机↔备机 | 受限 | ❌ | ❌ | ❌ | ✅ |
| 备机↔主机 | 部分 | ❌ | ✅ | ❌ | ✅ |
| 备机↔备机 | 最小 | ❌ | ❌ | ❌ | ✅ |

## 非对称切换

| 系统 | 超时 | 说明 |
|------|------|------|
| CTC | 3秒 | 较短，优先切换 |
| 联锁 | 5秒 | 较长，避免同时切换 |

## 其他通道故障

仅主机-主机故障时触发切换。其他三路故障仅报警，不倒机：

- 主机↔备机故障 → 失去监控，报警
- 备机↔主机故障 → 备机无法同步，报警
- 备机↔备机故障 → 失去状态交换，报警
