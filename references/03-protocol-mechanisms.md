## 超时与重传机制

### ACK超时值（固定不可修改）

⚠️ **协议约束：500ms超时是刚性限制，不可修改**

> 500ms超时值是**不可协商的协议标准**。在任何情况下都不能增加或减少。网络可靠性问题必须通过物理层改进解决，不能通过调整超时参数。

**500ms固定不变的原因：**
| 原因 | 说明 |
|------|------|
| 互操作性 | 确保不同CTC/CBI系统间的兼容 |
| 防止竞态 | 避免状态不一致 |
| 安全关键系统 | 需要确定性行为 |
| 协议合规 | 任何偏差都违反协议规范 |

### 超时参数表

| 参数 | 值 | 固定/可配置 | 说明 |
|------|------|-------------------|------|
| **数据帧ACK超时** | **500ms** | **固定** | 发送数据帧后等待ACK的时间 |
| **最大重试次数** | 3 | 固定 | 原始帧 + 2次重传 |
| **通信中断判定** | 1500ms | 推导值 | 3次失败 × 500ms |
| **硬件故障重试** | 6秒 | 固定 | 故障模式下的DC2发送间隔 |
| **SDIQ间隔** | 24小时 | 固定 | 每日同步周期 |
| **TSQ间隔** | 1小时 | 固定 | 每小时时间同步（除00:00外） |
| **心跳间隔** | 500ms | 固定 | 空闲期间ACK发送间隔 |

### 重传处理流程

```
发送方时序图：
  T=0ms     发送数据帧（发送序号=10）
  T=500ms   超时！未收到ACK
  T=500ms   重发相同帧（第2次尝试）
  T=1000ms  超时！仍未收到ACK
  T=1000ms  重发相同帧（第3次尝试）
  T=1500ms  超时！3次尝试均失败
  T=1500ms  → 判定通信中断
```

**重传要求：**
| 字段 | 是否允许修改 | 备注 |
|------|---------------|------|
| **发送序号** | ❌ 否 | 必须与原始帧相同 |
| **数据内容** | ❌ 否 | 必须完全相同 |
| **确认序号** | ✅ 是 | 更新为最近收到的帧 |
| **CRC** | ✅ 是 | 确认序号变化时需重新计算 |

### 通信中断检测条件

| 条件类型 | 判定条件 |
|----------|----------|
| **条件A** | 连续3次发送无确认响应（约1500ms） |
| **条件B** | 1500ms内未收到任何有效帧 |

### 中断后的处理措施

| 系统 | 处理措施 |
|------|----------|
| **CTC** | 发送DC2最多3次；若未收到DC3 → 判定硬件故障 |
| **联锁** | 停止发送数据，等待CTC重新初始化 |

### 硬件故障恢复

- CTC每6秒发送一次DC2尝试恢复
- 持续尝试直至连接建立或需要人工干预

## 序号控制机制

### 序号变量定义

双方系统各维护两个序号变量：

| 变量 | 用途 | 范围 | 初始值 |
|------|------|------|--------|
| **发送序号变量** | 为数据传输帧编号 | 0x00~0xFF | DC3后置为1 |
| **接收确认序号变量** | 校验收到的帧 | 0x00~0xFF | DC3后置为0 |

### 序号控制原则

#### 1. DC2/DC3连接帧不受序号控制

- 发送序号和确认序号均为0x00
- 不进行序号检查
- 仅用于连接建立

#### 2. DC3握手后的序号初始化

| 角色 | 发送序号变量 | 接收确认序号变量 |
|------|--------------|------------------|
| DC3发送方（联锁） | 置为1 | 置为0 |
| DC3接收方（CTC） | 置为1 | 置为0 |

#### 3. 发送数据帧时的序号使用

1. 将发送序号变量的值复制到帧的发送序号字段
2. 将接收确认序号变量的值复制到帧的确认序号字段
3. 必须收到确认后才能发送下一个数据帧

```
发送方状态：
  发送序号变量 = 10
  接收确认序号变量 = 5

待发送帧：
  发送序号 = 10（来自发送序号变量）
  确认序号 = 5（来自接收确认序号变量）
```

#### 4. 无确认不发送新数据规则

| 状态 | 处理措施 |
|------|----------|
| 收到ACK | 发送下一帧，序号递增 |
| 未收到ACK | 等待，不发送新数据帧 |
| 超时 | 重发相同帧或报告错误 |

#### 5. ACK确认验证（发送方）

将收到的确认序号与发送序号变量对比：

| 对比结果 | 解释 | 处理 |
|----------|------|------|
| **相等** | 帧已被确认 | 发送序号变量递增1 |
| **不相等** | 帧未被确认 | 丢弃，可能需要重传 |

```
发送方发送了发送序号=10的帧
发送方的发送序号变量=10（等待确认）

收到确认序号=10的帧：
  → 确认序号(10) == 发送序号变量(10) ✓
  → 上一帧已确认
  → 发送序号变量++（变为11）

收到确认序号=9的帧：
  → 确认序号(9) != 发送序号变量(10) ✗
  → 上一帧未被确认
```

#### 6. 序号连续性检查（接收方）

将收到的发送序号与接收确认序号变量对比：

| 差值 | 解释 | 处理 |
|------|------|------|
| **=1** | 预期帧到达 | 接收确认序号变量递增1，处理数据 |
| **=0** | 重复帧 | 发送ACK，**不重复处理数据** |
| **≥2** | 帧丢失 | 判定通信中断 |

**重复帧处理：**
```
接收方的接收确认序号变量=10

收到发送序号=10的帧：
  → 发送序号(10) == 接收确认序号变量(10)
  → 这是重复帧
  → 发送确认序号=10的ACK
  → 不再处理数据
```

**通信中断判定：**
```
接收方的接收确认序号变量=10

收到发送序号=12的帧：
  → 发送序号(12) - 接收确认序号变量(10) = 2
  → 序号11的帧已丢失！
  → 判定通信中断
  → 若为CTC：发送DC2重启
  → 若为联锁：停止并等待DC2
```

## 主备通信机制

### 通信路径矩阵

| 系统 | 角色1 | 角色2 |
|------|--------|--------|
| **CTC** | 主机（Host） | 备机（Standby） |
| **CBI** | 主机（Host） | 备机（Standby） |

### 各路径支持的帧类型

#### 1. CTC主机 → 联锁主机（完整通信）

**帧类型：** DC2, ACK, NACK, VERROR, SDIQ, BCC, TSD, RSR, ACA

**功能：** 连接建立、通信控制、数据查询和命令、时间同步、状态报告

#### 2. 联锁主机 → CTC主机（完整通信）

**帧类型：** DC3, ACK, NACK, VERROR, FIR, SDI, SDCI, TSQ, RSR, ACQ

**功能：** 连接确认、数据报告和请求、状态更新

#### 3. CTC主机 → 联锁备机（受限通信）

**帧类型：** DC2, ACK, NACK, VERROR, RSR

**功能：** 仅状态监控、无控制命令或数据查询

#### 4. 联锁备机 → CTC主机（受限通信）

**帧类型：** DC3, ACK, NACK, VERROR, RSR

**功能：** 仅状态报告、无站场数据或故障信息

#### 5. CTC备机 → 联锁主机（部分通信）

**帧类型：** DC2, ACK, NACK, VERROR, SDIQ, RSR

**功能：** 允许数据查询、**禁止控制命令**（BCC, TSD, ACA）

#### 6. 联锁主机 → CTC备机（部分通信）

**帧类型：** DC3, ACK, NACK, VERROR, SDI, SDCI, RSR

**功能：** 允许数据报告、**禁止时间查询**（TSQ）或自律请求（ACQ）

#### 7-8. 备机 ↔ 备机（最小通信）

**帧类型：** DC2, ACK, NACK, VERROR, RSR

**功能：** 仅基本状态交换、无数据查询或命令

### 主备传输汇总表

| 发送方 | 接收方 | 帧类型 | 控制命令 |
|--------|----------|-------------|------------------|
| CTC主机 | 联锁主机 | 完整 | ✅ 允许 |
| 联锁主机 | CTC主机 | 完整 | N/A（报告） |
| CTC主机 | 联锁备机 | 受限 | ❌ 禁止 |
| 联锁备机 | CTC主机 | 受限 | ❌ 禁止 |
| CTC备机 | 联锁主机 | 部分 | ❌ 禁止 |
| 联锁主机 | CTC备机 | 部分 | N/A（无请求） |
| 双方备机 | 双方备机 | 最小 | ❌ 禁止 |

## 通信故障切换机制

### 切换原则

**仅当主机-主机通信失败时才进行切换。** 其他路径故障触发告警但不进行切换。

### 备机同步要求

在与对端系统通信前，备机必须先完成内部同步：
- **CTC备机：** 与CTC主机同步数据
- **联锁备机：** 与联锁主机同步状态

### 切换触发条件

| 通信路径 | 故障响应 |
|--------------------|------------------|
| **CTC主机 ↔ 联锁主机** | ✅ **允许切换**（超时后） |
| CTC主机 ↔ 联锁备机 | ❌ 仅告警，不切换 |
| CTC备机 ↔ 联锁主机 | ❌ 仅告警，不切换 |
| 双方备机 ↔ 双方备机 | ❌ 仅告警，不切换 |

### CTC切换（3秒超时）

**触发条件：**
- 主机-主机通信中断超过3秒
- CTC备机可正常与联锁主机通信

```
CTC主机通信失败
         |
         ▼
   [启动定时器: 3秒]
         |
         ▼
   [3秒到？] ──否──► 继续监控
         | 是
         ▼
   [检查CTC备机 ──► 能否与
    ↔ 联锁主机]     联锁主机通信？
                        |
                 是    |    否
                  ↓      ↓
           [切换]      [不切换]
           CTC备机      [继续使用
           成为主机      CTC主机]
```

### 联锁切换（5秒超时）

**触发条件：**
- 主机-主机通信中断超过5秒
- 联锁备机可正常与CTC主机通信

```
联锁主机通信失败
         |
         ▼
   [启动定时器: 5秒]
         |
         ▼
   [5秒到？] ──否──► 继续监控
         | 是
         ▼
   [检查联锁备机 ──► 能否与
    ↔ CTC主机]       CTC主机通信？
                        |
                 是    |    否
                  ↓      ↓
           [切换]      [不切换]
           联锁备机     [继续使用
           成为主机     联锁主机]
```

### 切换时序对比

| 系统 | 超时时间 | 非对称设计原因 |
|------|---------|-------------------|
| **CTC** | 3秒 | 较短超时创建优先级 |
| **联锁** | 5秒 | 防止同时切换 |

### 安全要求

**切换前：**
1. ✅ 备机与主机完全同步
2. ✅ 备机与对端有活跃通信
3. ✅ 确认主机-主机通信失败

**切换期间：**
1. 平滑状态过渡
2. 序号同步
3. 恢复通信

**切换后：**
1. 原主机变为备机
2. 新主机接管所有主机功能
3. 监控原主机恢复情况
