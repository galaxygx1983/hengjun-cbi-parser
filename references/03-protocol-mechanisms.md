## 一、启动和建立连接

### 1.1 系统架构

CTC调度集中系统配置两台自律机互为主备，CBI计算机联锁系统配置两台操作表示机互为主备。正常状态下，双方维持交叉通信，共4路通信通道。

### 1.2 自律机启动流程

自律机启动后，首先判断自身是主机还是备机，决定后续动作。

#### 1.2.1 如果是主机

**第一步：同时与联锁机1、联锁机2建立通信**

自律机主机作为主动方，分别向联锁机1和联锁机2发起连接。每路连接的建立过程如下：

1. 自律机主机发送DC2帧（0x12）请求建立连接。
2. 联锁机收到DC2后，回复DC3帧（0x13）确认连接。
3. DC2/DC3握手完成后，双方序号初始化：发送序号变量置为1，接收确认序号变量置为0。

```
自律机主机                联锁机（1或2）
    |-------- DC2 (0x12) -------->|
    |<------- DC3 (0x13) ---------|
    |                             |
    [连接建立，序号初始化]
```

**第二步：判断联锁机谁是主机、谁是备机**

连接建立后，双方互发RSR帧（0xAA）交换工作状态：

- RSR帧包含两个关键字段：**主备状态**和**控制模式**。
- 主备状态值：0x55表示主机，0xAA表示备机，0xCC表示未知。

```
自律机主机                联锁机（1或2）
    |-------- RSR -------->|   （报告自律机主备状态）
    |<------- RSR ---------|   （报告联锁机主备状态）
    |                      |
    [识别对方主备身份]
```

自律机主机通过RSR帧中的主备状态值（0x55/0xAA），判断联锁机1和联锁机2分别是主机还是备机。

**第三步：初始数据同步（仅与联锁主机之间）**

识别出联锁主机后，联锁主机向自律机主机发送SDI帧（0x85），包含完整的站场状态数据，建立数据基线。自律机主机收到后回复ACK帧（0x06）确认。

```
自律机主机                联锁主机
    |<------- SDI (0x85) ---------|   （完整站场数据）
    |-------- ACK (0x06) -------->|   （确认收到）
    |                             |
    [数据基线建立完成]
```

**第四步：保持与联锁机1、联锁机2的通信**

自律机主机同时维持与联锁主机和联锁备机的通信，但两路通信的内容不同：

| 通信路径 | 通信级别 | 支持的帧类型 |
|----------|----------|-------------|
| 自律机主机 → 联锁主机 | **完整通信** | DC2, ACK, NACK, VERROR, SDIQ, BCC, TSD, RSR, ACA |
| 联锁主机 → 自律机主机 | **完整通信** | DC3, ACK, NACK, VERROR, FIR, SDI, SDCI, TSQ, RSR, ACQ |
| 自律机主机 → 联锁备机 | **受限通信** | DC2, ACK, NACK, VERROR, RSR |
| 联锁备机 → 自律机主机 | **受限通信** | DC3, ACK, NACK, VERROR, RSR |

核心区别：与联锁主机之间可以进行控制命令（BCC）、站场数据（SDI/SDCI）、时间同步（TSQ/TSD）、自律请求（ACQ/ACA）等完整交互；与联锁备机之间仅进行状态监控（RSR）和连接维护。

#### 1.2.2 如果是备机

**第一步：与自律机主机建立连接，完成内部同步**

自律机备机启动后，首先与自律机主机建立内部连接，同步数据和状态。备机必须先完成与主机的数据同步，才能与外部联锁机通信。

**第二步：向主机发起请求，连接联锁机**

自律机备机向自律机主机发起请求，申请与联锁机建立通信。主机同意后，备机同时与联锁机1和联锁机2建立连接。

建立连接的过程与主机相同：

1. 自律机备机分别向联锁机1和联锁机2发送DC2帧（0x12）。
2. 联锁机回复DC3帧（0x13）确认连接。
3. 双方互发RSR帧，判断联锁机的主备身份。

**第三步：保持与联锁机1、联锁机2的通信**

自律机备机同时维持与联锁主机和联锁备机的通信，但通信内容与自律机主机不同：

| 通信路径 | 通信级别 | 支持的帧类型 |
|----------|----------|-------------|
| 自律机备机 → 联锁主机 | **部分通信** | DC2, ACK, NACK, VERROR, SDIQ, RSR |
| 联锁主机 → 自律机备机 | **部分通信** | DC3, ACK, NACK, VERROR, SDI, SDCI, RSR |
| 自律机备机 → 联锁备机 | **最小通信** | DC2, ACK, NACK, VERROR, RSR |
| 联锁备机 → 自律机备机 | **最小通信** | DC3, ACK, NACK, VERROR, RSR |

核心区别：自律机备机与联锁主机之间允许数据查询（SDIQ）和接收站场数据（SDI/SDCI），但**禁止发送控制命令**（BCC、TSD、ACA）；与联锁备机之间仅进行基本状态交换。

### 1.3 最终状态：4路通信矩阵

自律机与联锁机之间同时保持4路通信，每路通信交互的帧类型有明确区别：

| 通信路径 | 通信级别 | 控制命令 | 站场数据 | 状态报告 |
|----------|----------|----------|----------|----------|
| 自律机主机 ↔ 联锁主机 | 完整通信 | ✅ 允许（BCC/ACA/TSD） | ✅ 完整（SDI/SDCI/FIR/SDIQ/TSQ/ACQ） | ✅ RSR |
| 自律机主机 ↔ 联锁备机 | 受限通信 | ❌ 禁止 | ❌ 无 | ✅ RSR |
| 自律机备机 ↔ 联锁主机 | 部分通信 | ❌ 禁止 | ✅ 可查询（SDIQ→SDI/SDCI） | ✅ RSR |
| 自律机备机 ↔ 联锁备机 | 最小通信 | ❌ 禁止 | ❌ 无 | ✅ RSR |

### 1.4 连接维持机制

所有4路通信均通过以下机制维持连接健康：

- **心跳机制**：无数据收发超过500ms时，发送ACK帧（0x06）作为心跳保活信号。
- **通信中断检测**：1500ms内未收到任何有效帧，判定该路通信中断。
- **中断后处理**：自律机发送DC2帧最多3次尝试恢复；若3次均未收到DC3，判定为"硬件故障"，进入硬件故障处理模式，每6秒发送一次DC2尝试恢复。

## 二、超时与重传机制

### 2.1 ACK超时值（固定不可修改）

⚠️ **协议约束：500ms超时是刚性限制，不可修改**

> 500ms超时值是**不可协商的协议标准**。在任何情况下都不能增加或减少。网络可靠性问题必须通过物理层改进解决，不能通过调整超时参数。

**500ms固定不变的原因：**
| 原因 | 说明 |
|------|------|
| 互操作性 | 确保不同CTC/CBI系统间的兼容 |
| 防止竞态 | 避免状态不一致 |
| 安全关键系统 | 需要确定性行为 |
| 协议合规 | 任何偏差都违反协议规范 |

### 2.2 超时参数表

| 参数 | 值 | 固定/可配置 | 说明 |
|------|------|-------------------|------|
| **数据帧ACK超时** | **500ms** | **固定** | 发送数据帧后等待ACK的时间 |
| **最大重试次数** | 3 | 固定 | 原始帧 + 2次重传 |
| **通信中断判定** | 1500ms | 推导值 | 3次失败 × 500ms |
| **硬件故障重试** | 6秒 | 固定 | 故障模式下的DC2发送间隔 |
| **SDIQ间隔** | 24小时 | 固定 | 每日同步周期 |
| **TSQ间隔** | 1小时 | 固定 | 每小时时间同步（除00:00外） |
| **心跳间隔** | 500ms | 固定 | 空闲期间ACK发送间隔 |
| **ACA响应超时** | 5秒 | 固定 | 联锁发送ACQ后等待CTC回复ACA的时间 |

### 2.3 重传处理流程

```
发送方时序图：
  T=0ms     发送数据帧（发送序号=10）
  T=500ms   超时！未收到ACK
  T=500ms   重发相同帧（第2次尝试）
  T=1000ms  超时！仍未收到ACK
  T=1000ms  重发相同帧（第3次尝试）
  T=1500ms  超时！3次尝试均失败
  T=1500ms  → 判定通信中断
```

**重传要求：**
| 字段 | 是否允许修改 | 备注 |
|------|---------------|------|
| **发送序号** | ❌ 否 | 必须与原始帧相同 |
| **数据内容** | ❌ 否 | 必须完全相同 |
| **确认序号** | ✅ 是 | 更新为最近收到的帧 |
| **CRC** | ✅ 是 | 确认序号变化时需重新计算 |

## 三、序号控制机制

### 3.1 序号变量定义

双方系统各维护两个序号变量：

| 变量 | 用途 | 范围 | 初始值 |
|------|------|------|--------|
| **发送序号变量** | 为数据传输帧编号 | 0x00~0xFF | DC3后置为1 |
| **接收确认序号变量** | 校验收到的帧 | 0x00~0xFF | DC3后置为0 |

### 3.2 序号控制原则

#### 1. DC2/DC3连接帧不受序号控制

- 发送序号和确认序号均为0x00
- 不进行序号检查
- 仅用于连接建立

#### 2. DC3握手后的序号初始化

| 角色 | 发送序号变量 | 接收确认序号变量 |
|------|--------------|------------------|
| DC3发送方（联锁） | 置为1 | 置为0 |
| DC3接收方（CTC） | 置为1 | 置为0 |

#### 3. 发送数据帧时的序号使用

1. 将发送序号变量的值复制到帧的发送序号字段
2. 将接收确认序号变量的值复制到帧的确认序号字段
3. 必须收到确认后才能发送下一个数据帧

```
发送方状态：
  发送序号变量 = 10
  接收确认序号变量 = 5

待发送帧：
  发送序号 = 10（来自发送序号变量）
  确认序号 = 5（来自接收确认序号变量）
```

#### 4. 无确认不发送新数据规则

| 状态 | 处理措施 |
|------|----------|
| 收到ACK | 发送下一帧，序号递增 |
| 未收到ACK | 等待，不发送新数据帧 |
| 超时 | 重发相同帧或报告错误 |

#### 5. ACK确认验证（发送方）

将收到的确认序号与发送序号变量对比：

| 对比结果 | 解释 | 处理 |
|----------|------|------|
| **相等** | 帧已被确认 | 发送序号变量递增1 |
| **不相等** | 帧未被确认 | 丢弃，可能需要重传 |

```
发送方发送了发送序号=10的帧
发送方的发送序号变量=10（等待确认）

收到确认序号=10的帧：
  → 确认序号(10) == 发送序号变量(10) ✓
  → 上一帧已确认
  → 发送序号变量++（变为11）

收到确认序号=9的帧：
  → 确认序号(9) != 发送序号变量(10) ✗
  → 上一帧未被确认
```

#### 6. 序号连续性检查（接收方）

将收到的发送序号与接收确认序号变量对比：

| 差值 | 解释 | 处理 |
|------|------|------|
| **=1** | 预期帧到达 | 接收确认序号变量递增1，处理数据 |
| **=0** | 重复帧 | 发送ACK，**不重复处理数据** |
| **≥2** | 帧丢失 | 判定通信中断 |

**重复帧处理：**
```
接收方的接收确认序号变量=10

收到发送序号=10的帧：
  → 发送序号(10) == 接收确认序号变量(10)
  → 这是重复帧
  → 发送确认序号=10的ACK
  → 不再处理数据
```

**通信中断判定：**
```
接收方的接收确认序号变量=10

收到发送序号=12的帧：
  → 发送序号(12) - 接收确认序号变量(10) = 2
  → 序号11的帧已丢失！
  → 判定通信中断
  → 若为CTC：发送DC2重启
  → 若为联锁：停止并等待DC2
```

## 四、主备通信机制

### 4.1 通信路径矩阵

| 系统 | 角色1 | 角色2 |
|------|--------|--------|
| **CTC** | 主机（Host） | 备机（Standby） |
| **CBI** | 主机（Host） | 备机（Standby） |

### 4.2 各路径支持的帧类型

| 帧类型 | 编码 | 主机↔联锁主机 | 主机↔联锁备机 | 备机↔联锁主机 | 备机↔联锁备机 |
|--------|------|:---:|:---:|:---:|:---:|
| DC2（连接请求） | 0x12 | ✅ | ✅ | ✅ | ✅ |
| DC3（连接确认） | 0x13 | ✅ | ✅ | ✅ | ✅ |
| ACK（应答/心跳） | 0x06 | ✅ | ✅ | ✅ | ✅ |
| NACK（否定应答） | 0x15 | ✅ | ✅ | ✅ | ✅ |
| VERROR（版本错误） | 0x10 | ✅ | ✅ | ✅ | ✅ |
| RSR（状态报告） | 0xAA | ✅ | ✅ | ✅ | ✅ |
| SDI（完整站场数据） | 0x85 | ✅ | ❌ | ✅ | ❌ |
| SDCI（增量站场数据） | 0x8A | ✅ | ❌ | ✅ | ❌ |
| SDIQ（站场数据请求） | 0x6A | ✅ | ❌ | ✅ | ❌ |
| FIR（故障信息报告） | 0x65 | ✅ | ❌ | ❌ | ❌ |
| BCC（控制命令） | 0x95 | ✅ | ❌ | ❌ | ❌ |
| TSQ（时间同步请求） | 0x9A | ✅ | ❌ | ❌ | ❌ |
| TSD（时间同步数据） | 0xA5 | ✅ | ❌ | ❌ | ❌ |
| ACQ（自律请求） | 0x75 | ✅ | ❌ | ❌ | ❌ |
| ACA（自律同意） | 0x7A | ✅ | ❌ | ❌ | ❌ |

### 4.3 主备传输汇总表

| 发送方 | 接收方 | 帧类型 | 控制命令 |
|--------|----------|-------------|------------------|
| CTC主机 | 联锁主机 | 完整 | ✅ 允许 |
| 联锁主机 | CTC主机 | 完整 | N/A（报告） |
| CTC主机 | 联锁备机 | 受限 | ❌ 禁止 |
| 联锁备机 | CTC主机 | 受限 | ❌ 禁止 |
| CTC备机 | 联锁主机 | 部分 | ❌ 禁止 |
| 联锁主机 | CTC备机 | 部分 | N/A（无请求） |
| 双方备机 | 双方备机 | 最小 | ❌ 禁止 |

## 五、通信故障切换机制

### 5.1 切换原则

**仅当主机-主机通信失败时才进行切换。** 其他路径故障触发告警但不进行切换。

### 5.2 备机同步要求

在与对端系统通信前，备机必须先完成内部同步：
- **CTC备机：** 与CTC主机同步数据
- **联锁备机：** 与联锁主机同步状态

### 5.3 切换触发条件

| 通信路径 | 故障响应 |
|--------------------|------------------|
| **CTC主机 ↔ 联锁主机** | ✅ **允许切换**（超时后） |
| CTC主机 ↔ 联锁备机 | ❌ 仅告警，不切换 |
| CTC备机 ↔ 联锁主机 | ❌ 仅告警，不切换 |
| 双方备机 ↔ 双方备机 | ❌ 仅告警，不切换 |

### 5.4 硬件故障判定流程

当自律机主机与联锁主机之间出现通信异常时，按以下流程逐步升级：

```
正常通信中
    |
    ▼
[检测到通信超时]（1500ms内未收到有效帧，或连续3次发送无ACK确认）
    |
    ▼
[进入"通信中断"状态]
    |
    ▼
[发送DC2帧尝试恢复连接]（最多3次，每次等待DC3响应）
    |
    ├── 收到DC3 → 连接恢复，重新进行RSR状态交换和SDI数据同步
    |
    └── 3次DC2均未收到DC3
         |
         ▼
    [判定为"硬件故障"]
         |
         ▼
    [进入硬件故障处理模式]（每6秒发送一次DC2尝试恢复）
```

### 5.5 CTC切换（3秒超时）

**触发条件：**
- 主机-主机通信中断超过3秒
- CTC备机可正常与联锁主机通信

```
CTC主机通信失败
         |
         ▼
   [启动定时器: 3秒]
         |
         ▼
   [3秒到？] ──否──► 继续监控
         | 是
         ▼
   [检查CTC备机 ──► 能否与
    ↔ 联锁主机]     联锁主机通信？
                        |
                 是    |    否
                  ↓      ↓
           [切换]      [不切换]
           CTC备机      [继续使用
           成为主机      CTC主机]
```

**情况一：CTC备机与联锁主机通信不正常**

此时自律机侧无法通过倒机恢复与联锁主机的通信（因为备机与联锁主机的通信也不正常），自律机不做任何倒机操作，等待联锁侧的倒机处理。

具体流程：

1. 自律机主机持续在硬件故障处理模式下，每6秒向联锁主机发送DC2帧尝试恢复。
2. 联锁侧检测到与自律机主机通信中断超过5秒后（联锁切换超时为5秒），联锁进行倒机：原联锁备机升级为联锁主机。
3. 新的联锁主机（原备机）通过RSR帧通知自律机主机其已升级为主机。
4. 自律机主机收到新联锁主机的DC3响应后，重新建立连接，进行RSR状态交换和SDI数据同步，恢复双方通信。

```
自律机主机          联锁主机（故障）      联锁备机
    |                    |                   |
    |-- DC2（每6秒）---->|（无响应）          |
    |                    |                   |
    |   [等待联锁倒机]   |   [5秒超时到]     |
    |                    |                   |
    |                    |    [联锁备机升级为主机]
    |                    |                   |
    |<------------- DC3 --------------------|  （新联锁主机响应）
    |-------------- RSR ------------------->|  （状态交换）
    |<------------- RSR --------------------|
    |<------------- SDI --------------------|  （数据同步）
    |-------------- ACK ------------------->|
    |                                       |
    [通信恢复]
```

**情况二：CTC备机与联锁主机通信正常**

此时可以通过自律机侧倒机恢复与联锁主机的通信。自律机进行倒机：原自律机备机升级为主机，与联锁主机进行通信，恢复双方通信。

具体流程：

1. 自律机主机检测到与联锁主机硬件故障持续3秒，同时确认自律机备机与联锁主机通信正常。
2. 自律机进行倒机：原自律机主机降级为备机，原自律机备机升级为主机。
3. 新的自律机主机（原备机）已与联锁主机保持通信（部分通信级别），升级后切换为完整通信级别。
4. 新自律机主机与联锁主机互发RSR帧，报告新的主备状态。
5. 联锁主机向新自律机主机发送SDI帧进行完整数据同步（如需要），恢复双方通信。

```
自律机主机（故障中）    自律机备机           联锁主机
    |                      |                   |
    |  [3秒超时到]         |                   |
    |  [确认备机与联锁     |<--- 通信正常 ---->|
    |   主机通信正常]      |                   |
    |                      |                   |
    [降级为备机]     [升级为主机]              |
                           |                   |
                           |------ RSR ------->|  （报告新主备状态）
                           |<----- RSR --------|
                           |<----- SDI --------|  （数据同步）
                           |------ ACK ------->|
                           |                   |
                           [通信恢复，完整通信级别]
```

### 5.6 联锁切换（5秒超时）

**触发条件：**
- 主机-主机通信中断超过5秒
- 联锁备机可正常与CTC主机通信

```
联锁主机通信失败
         |
         ▼
   [启动定时器: 5秒]
         |
         ▼
   [5秒到？] ──否──► 继续监控
         | 是
         ▼
   [检查联锁备机 ──► 能否与
    ↔ CTC主机]       CTC主机通信？
                        |
                 是    |    否
                  ↓      ↓
           [切换]      [不切换]
           联锁备机     [继续使用
           成为主机     联锁主机]
```

### 5.7 CTC与联锁切换超时的非对称设计

| 系统 | 切换超时 | 设计原因 |
|------|----------|----------|
| CTC自律机 | **3秒** | 较短超时，优先由CTC侧切换 |
| CBI联锁机 | **5秒** | 较长超时，避免双方同时切换 |

这种非对称设计确保：当主机-主机通信故障时，CTC侧（3秒）先于联锁侧（5秒）做出切换判断，避免双方同时倒机导致的状态混乱。

### 5.8 安全要求

**切换前：**
1. ✅ 备机与主机完全同步
2. ✅ 备机与对端有活跃通信
3. ✅ 确认主机-主机通信失败

**切换期间：**
1. 平滑状态过渡
2. 序号同步
3. 恢复通信

**切换后：**
1. 原主机变为备机
2. 新主机接管所有主机功能
3. 监控原主机恢复情况

## 六、其他通道通信故障处理

当自律机主机与联锁主机通信正常，而另外三个通道中任何一个出现通信故障时：

- **仅报警，不做任何倒机操作。**

原因：主机-主机通道是核心控制通道，只要该通道正常，系统即可正常运行。其他通道的故障不影响核心控制功能，仅影响冗余监控能力，因此只需报警提示运维人员排查，无需触发倒机。

具体报警场景：

| 故障通道 | 影响范围 | 处理方式 |
|----------|----------|----------|
| 自律机主机 ↔ 联锁备机 | 失去对联锁备机的状态监控 | 报警，不倒机 |
| 自律机备机 ↔ 联锁主机 | 备机无法同步联锁主机数据 | 报警，不倒机 |
| 自律机备机 ↔ 联锁备机 | 失去备机间状态交换 | 报警，不倒机 |

## 七、故障处理决策树

```
检测到某路通信故障
    |
    ▼
[故障通道是否为"自律机主机 ↔ 联锁主机"？]
    |
    ├── 否 → 仅报警，不做任何倒机操作
    |
    └── 是
         |
         ▼
    [故障类型判断]
         |
         ├── 序号交互异常 → 按序号异常处理规则处理
         |
         └── 硬件故障（持续3秒无法恢复）
              |
              ▼
         [自律机备机与联锁主机通信是否正常？]
              |
              ├── 正常 → 自律机倒机（备机升级为主机，恢复通信）
              |
              └── 不正常 → 自律机不操作，等待联锁倒机
                           （联锁备机5秒后升级为主机，通过RSR通知自律机）
```

## 八、网络问题处理原则

网络问题必须通过物理层改善解决，不能通过调整超时值解决。

| 正确做法 | 错误做法 |
|----------|----------|
| 检查电缆连接 | 增加ACK超时值 |
| 验证接地系统 | 修改重试次数 |
| 测试信号质量 | 调整心跳间隔 |
| 更换网络设备 | 修改切换超时 |
