## 设备状态变化分析

分析CTC日志中的设备状态变化，包括信号机、道岔、区段等设备的状态切换。

### 日志格式

CTC日志中设备状态变化以以下格式记录：

```
2026-02-11 04:19:47.414 [INFO][7fb313fff700][0][ApplySDCIToSDIHJ:2441]
=== Begin Change ===
Device Index: 91
Config Type: 2, SDI Byte Index: 127
Old State: 0x02 (2)
New State: 0x04 (4)
2026-02-11 04:19:47.414 [INFO][7fb313fff700][0][ApplySDCIToSDIHJ:2464] Updated State: 0x04 (4)
=== End Change ===
```

### 字段说明

| 字段 | 说明 | 示例 |
|------|------|------|
| `Device Index` | 设备在objects表中的索引 | 91 (对应D160) |
| `Config Type` | 配置类型 (1=objects, 2=zlobjects) | 2 |
| `SDI Byte Index` | SDI缓冲区字节索引 (从1开始) | 127 |
| `Old State` | 状态变化前的值 | 0x02 |
| `New State` | 状态变化后的值 | 0x04 |

### 设备索引查询

设备索引存储在 `lgxtq.zl` 文件的 `[objects]` 表中：

```bash
# 查询D160的设备索引
grep "D160" lgxtq.zl
# 输出: #,D160,91
```

**关键字段映射：**
- `objects` 表：`#,设备名,索引号` → 用于SDCI帧解析
- `zlobjects` 表：`#,设备名,字节索引,位索引` → 用于SDI缓冲区映射

### 状态解码

#### 信号机状态 (Bit 0-7)

| 位 | 值 | 含义 |
|----|-----|------|
| Bit 0 | 0x01 | 灯丝断丝 |
| Bit 1 | 0x02 | 蓝灯 (调车禁止) |
| Bit 2 | 0x04 | 白灯 (调车允许) |
| Bit 3 | 0x08 | 红灯 (停车) |
| Bit 4 | 0x10 | 绿灯 (开放正线) |
| Bit 5 | 0x20 | 黄/2黄/引白组合 |
| Bit 6 | 0x40 | 进路转岔中 |
| Bit 7 | 0x80 | 延时解锁 |

#### 信号机典型状态值

| 状态值 | 二进制 | 灯色 | 说明 |
|--------|--------|------|------|
| 0x00 | 00000000 | 灭灯 | 无显示 |
| 0x02 | 00000010 | 蓝灯 | 调车禁止 |
| 0x04 | 00000100 | 白灯 | 调车允许 |
| 0x08 | 00001000 | 红灯 | 停车 |
| 0x10 | 00010000 | 绿灯 | 开放正线 |
| 0x42 | 01000010 | 蓝灯+转岔 | 调车禁止+进路转岔中 |

#### 道岔区段状态 (Bit 0-7)

| 位 | 值 | 含义 |
|----|-----|------|
| Bit 0 | 0x01 | 定位 |
| Bit 1 | 0x02 | 反位 |
| Bit 2 | 0x04 | 无表示 |
| Bit 3 | 0x08 | 区段锁闭 |
| Bit 4 | 0x10 | 区段占用 |
| Bit 5 | 0x20 | 道岔锁闭 |

#### 无岔区段状态

| 位 | 值 | 含义 |
|----|-----|------|
| Bit 0/4 | 0x01/0x10 | 区段锁闭 |
| Bit 1/5 | 0x02/0x20 | 区段占用 |

### 分析示例

#### 信号机灯色变化分析

**日志模式识别：**
```bash
# 提取所有设备状态变化
grep -A 6 "=== Begin Change ===" lgxtcidriver_*.log | grep -E "Device Index:|Old State:|New State:"

# 筛选特定设备 (D160 = Device Index 91)
grep -B 3 -A 3 "Device Index: 91" lgxtcidriver_*.log
```

**状态转换分析：**
```python
# 示例：统计蓝灯白灯切换
from collections import defaultdict

changes = defaultdict(int)
for entry in device_changes:
    if entry['device_index'] == 91:  # D160
        if entry['old_state'] == '0x02' and entry['new_state'] == '0x04':
            changes['blue_to_white'] += 1
        elif entry['old_state'] == '0x04' and entry['new_state'] == '0x02':
            changes['white_to_blue'] += 1
```

#### 道岔状态变化分析

**典型状态值：**
| 状态值 | 占用 | 区段锁闭 | 道岔锁闭 | 位置 | 说明 |
|--------|------|----------|----------|------|------|
| 0x00 | 否 | 否 | 否 | 定位 | 空闲未锁 |
| 0x01 | 是 | 否 | 否 | 定位 | 列车通过 |
| 0x02 | 否 | 是 | 否 | 反位 | 进路锁闭 |
| 0x08 | 否 | 否 | 否 | 反位 | 空闲未锁 |
| 0x19 | 是 | 否 | 否 | 定位+无表示 | **位置异常** |

### 异常检测

**信号机异常状态：**
| 状态值 | 模式 | 可能原因 |
|--------|------|----------|
| 0x01 | 灯丝断丝 | 灯泡故障 |
| 0x42 | 蓝灯+进路转岔 | 转岔过程中开放信号 |
| 0x88 | 红灯+延时解锁 | 信号关闭后延时 |

**道岔异常状态：**
| 状态值 | 模式 | 可能原因 |
|--------|------|----------|
| 0x07 | 定位+反位+无表示 | 转辙机故障 |
| 0x19 | 定位+无表示+占用 | 位置丢失但列车占用 |

### 告警阈值

| 指标 | 正常范围 | 警告 | 严重 |
|------|----------|------|------|
| 四开状态频率 | <5% | 5-10% | >10% |
| 状态变化频率 | <10次/小时 | 10-30次/小时 | >30次/小时 |
| 占用持续时间 | <5分钟 | 5-10分钟 | >10分钟 |

## 硬件故障分析

分析CTC日志中的硬件故障和通信中断问题。

### DC2/DC3握手协议

| 帧类型 | 编码 | 说明 |
|--------|------|------|
| DC2 | 0x12 | 连接建立请求 |
| DC3 | 0x13 | 连接确认响应 |

### 故障检测序列

```
1. 检测到通信超时（超过1500ms未收到有效帧）
           |
           ▼
2. 系统进入"通信中断"状态
           |
           ▼
3. 发送DC2帧重试建立连接（最多3次）
           |
           ▼
4. 如果3次DC2后仍未收到DC3 → 判定为"硬件故障"
           |
           ▼
5. 系统进入硬件故障处理模式
           |
           ▼
6. 故障模式下，每6秒发送一次DC2尝试恢复
           |
           ▼
7. 系统尝试连接联锁备机
```

### CTC日志分析

**Python分析代码：**
```python
from hengjun_cbi_parser import CTCLogHardwareFaultAnalyzer

# 分析CTC日志中的硬件故障
analyzer = CTCLogHardwareFaultAnalyzer("ctc.log")
result = analyzer.analyze()

# 生成文本报告
analyzer.generate_report(output_file="hardware_fault_report.txt")

# 导出为CSV供进一步分析
analyzer.export_to_csv("hardware_fault_events.csv")

# 访问分析结果
print(f"故障事件总数: {result['statistics']['total_fault_events']}")
print(f"恢复事件数: {result['statistics']['total_recovery_events']}")
print(f"发送DC2帧数: {result['statistics']['total_dc2_frames']}")
```

### 故障事件分类

**故障事件：**
| 事件类型 | 严重级别 | 触发条件 |
|----------|----------|----------|
| `communication_interruption` | 警告 | 超过1500ms未收到有效帧 |
| `hardware_fault` | 错误 | 3次DC2失败后未收到DC3 |

**恢复事件：**
| 事件类型 | 说明 |
|----------|------|
| `DC3_RECEIVED` | 通过DC3帧确认连接 |
| `CONNECTION_RESTORED` | 链路成功恢复 |

### 日志模式识别

**通信中断：**
```
[WARNING] 检测到通信中断 (超过 1500 ms 未收到有效帧)
```

**硬件故障：**
```
[ERROR] 检测到硬件故障 (发送 3 次 DC2 后仍未收到 DC3)，尝试与联锁备机通信
```

**硬件故障处理模式：**
```
[INFO] 进入硬件故障处理模式，将每 6 秒发送一次 DC2
```

**DC2帧示例：**
```
[INFO] DC2帧内容: [7D 04 11 00 00 12 66 D6 7E]
```

### 预期结果

**典型故障模式：**
- 检测到通信中断（警告）
- 约2秒内发送3个DC2帧
- 判定为硬件故障（错误）
- 系统进入故障处理模式
- 尝试连接联锁备机
- 故障模式下每6秒发送DC2帧

**统计示例：**
```
故障事件总数: 10
  - 通信中断: 10
  - 硬件故障: 10
发送DC2帧总数: 30+
故障模式下DC2帧: 20+
平均故障间隔: ~900秒 (15分钟)
```

### 故障排查

| 症状 | 排查方向 |
|------|----------|
| 频繁中断 | 检查物理层（电缆、交换机） |
| 无DC3响应 | 验证联锁系统是否发送DC3 |
| 集中在特定线程 | 监控线程ID进行调试 |
| 故障间隔长 | 间歇性网络问题 |
| 需要关联分析 | 与其他系统事件对比时间戳 |

## 正常与异常重连分析

分析CTC驱动日志，区分正常重启（计划维护）和异常重连（需要主备切换的硬件故障）。

### 关键日志标记

| 事件类型 | 日志模式 | 说明 |
|----------|----------|------|
| **正常重启** | `驱动启动后首次读取，初始化连接` | 驱动进程重启，通常是计划维护 |
| **异常重连** | `检测到硬件故障 (发送 3 次 DC2 后仍未收到 DC3)` | 检测到硬件故障，自动切换到备机 |
| **硬件故障模式** | `进入硬件故障处理模式，将每 6 秒发送一次 DC2` | 系统处于故障恢复模式 |
| **连接成功** | `收到DC3，连接已建立` | DC2/DC3握手成功 |

### 正常重启模式

**特征：**
- 由驱动进程重启触发（计划维护）
- 日志显示：`驱动启动后首次读取，初始化连接`
- 通常紧跟DC2/DC3握手
- 无"硬件故障"错误消息
- 第一次或第二次尝试即连接成功

**日志序列：**
```
[INFO] 驱动启动后首次读取，初始化连接
[INFO] DC2帧内容: [7D 04 11 00 00 12 66 D6 7E]
[INFO] 收到DC3，连接已建立
```

**两种子场景：**

#### 场景A：联锁1号机为主机（直接连接）

当联锁1号机（联锁1号机）是当前主机时：

**日志序列：**
```
[INFO] 驱动启动后首次读取，初始化连接              ← 驱动重启
[INFO] DC2帧内容: [7D 04 11 00 00 12 66 D6 7E]   ← 连接1号机
[INFO] 收到DC3，连接已建立                        ← 立即成功
[INFO] RSR主备机状态值: 0x55                     ← 确认1号机为主机
```

**特征：**
- 单次DC2/DC3握手
- 立即连接成功
- RSR状态显示0x55（1号机为主机）
- 连接耗时：< 1秒

#### 场景B：联锁2号机为主机（自动切换）

当联锁2号机（联锁2号机）是当前主机时，CTC驱动先尝试1号机，然后自动切换到2号机：

**日志序列：**
```
[INFO] 驱动启动后首次读取，初始化连接              ← 驱动重启
[INFO] DC2帧内容: [7D 04 11 00 00 12 66 D6 7E]   ← 尝试1号机
[INFO] 收到DC3，连接已建立                        ← 连接到1号机
[INFO] RSR主备机状态值: 0xAA                     ← 检测到1号机是备机
[INFO] 检测到对方为备机，主动断开连接并重置状态   ← 从备机断开
[INFO] DC2帧内容: [7D 04 11 00 00 12 66 D6 7E]   ← 尝试2号机
[INFO] 收到DC3，连接已建立                        ← 连接到2号机
[INFO] RSR主备机状态值: 0x55                     ← 确认2号机为主机
```

**特征：**
- 两次DC2/DC3握手（1号机 → 断开 → 2号机）
- 第一次连接显示RSR=0xAA（备机）
- 系统自动断开并重试
- 第二次连接显示RSR=0x55（主机）
- 总连接耗时：1-2秒

**重要说明：**
这不是硬件故障！系统行为正确：
1. 尝试默认连接1号机
2. 通过RSR检测到1号机是备机（0xAA）
3. 主动断开并尝试2号机
4. 成功连接到实际主机（2号机）

**与异常重连的区分方法：**
- **正常（场景B）**：日志显示`检测到对方为备机，主动断开连接`（主动断开）
- **异常**：日志显示`检测到硬件故障`（3次DC2失败后判定硬件故障）

**分析要点：**
- 统计次数：`grep -c "驱动启动后首次读取" logfile`
- 重启间隔应合理（>几分钟）
- 成功率：~100%
- 检查RSR状态转换：0xAA → 断开 → 0x55 表示场景B

### 异常重连模式

**特征：**
- 由硬件故障检测触发
- 日志显示：`检测到硬件故障 (发送 3 次 DC2 后仍未收到 DC3)`
- 故障声明前多次DC2失败
- 系统进入"硬件故障处理模式"
- 自动切换到联锁备机

**日志序列：**
```
[ERROR] 检测到硬件故障 (发送 3 次 DC2 后仍未收到 DC3)，尝试与联锁备机通信
[INFO] 进入硬件故障处理模式，将每 6 秒发送一次 DC2
[INFO] DC2帧内容: [7D 04 11 00 00 12 66 D6 7E]  # 每6秒重试
[INFO] 收到DC3，连接已建立  # 连接备机成功
```

**分析要点：**
- 统计硬件故障次数：`grep -c "检测到硬件故障" logfile`
- 统计故障模式下的DC2重试次数
- 故障持续时间（从故障到恢复）
- 检查故障期间是否出现FIR帧

### 对比表

| 指标 | 正常重启 | 异常重连 |
|------|----------|----------|
| **触发原因** | 驱动重启（计划） | 硬件故障（自动） |
| **日志标记** | `驱动启动后首次读取` | `检测到硬件故障` |
| **DC2尝试次数** | 通常1-2次 | 故障声明前3+次 |
| **故障模式** | 否 | 是（6秒DC2间隔） |
| **主备切换** | 通常重连到同一主机 | 切换到备机 |
| **FIR帧** | 罕见 | 故障前/期间常出现 |
| **用户影响** | 短暂中断 | 可能服务降级 |

### 分析命令

**统计正常重启次数：**
```bash
grep "驱动启动后首次读取" lgxtcidriver_*.log
```

**统计硬件故障次数：**
```bash
grep "检测到硬件故障" lgxtcidriver_*.log
```

**分析重连时间线：**
```bash
# 提取所有重连事件
awk '/驱动启动后首次读取|检测到硬件故障|收到DC3.*连接已建立/' logfile | \
  awk '{print $1, $2, $0}'
```

**计算事件间时间间隔：**
```bash
# 从重启到下一事件的时间
awk '/驱动启动后首次读取/{t1=$1" "$2} /检测到硬件故障/{t2=$1" "$2} t1 && t2{print t1 " -> " t2; t1=""; t2=""}' logfile
```

```
14:40:29.365 [INFO] 连接备机 (RSR=0xAA)             ← 连接到备机
14:40:32.864 [ERROR] 检测到硬件故障...              ← 备机也故障
14:40:37.865 [INFO] 收到DC3，连接已建立              ← 重连到主机
14:48:36.412 [ERROR] 检测到硬件故障...              ← 主机再次故障
```
**评估：** 严重 - 两台主机都不稳定，需立即调查

### 故障排查指南

| 模式 | 可能原因 | 处理措施 |
|------|----------|----------|
| 频繁正常重启 | 自动重启配置错误 | 检查systemd/cron设置 |
| 每次重启后硬件故障 | 主机启动问题 | 检查联锁启动序列 |
| 交替主备故障 | 网络不稳定 | 检查物理连接 |
| 故障前FIR爆发 | 设备故障导致主机压力 | 检查FIR错误码 |
| 故障恢复时间长（>30秒） | 备机响应慢 | 检查备机健康状态 |
